Moteur GPO — Politiques de Groupe

Comment les politiques cascadent dans l'arbre organisationnel

Anatomie d'un objet Policy (GPO)

POLICY
pol_security_mfa_required Priorite: 100
Scope (ou s'applique)
• link_type: node | group | global
• linked_to: UUID du noeud/groupe
• enforced: true = non-blocable
• disabled: false
Domaine (quoi)
• security — auth, MFA, sessions
• modules — acces aux apps
• naming — conventions nommage
• delegation — qui administre quoi
• compliance — regles pays/legal
• custom — extensible JSONB
Settings (les regles)
{
  "auth.mfa_required": true,
  "auth.session_max_hours": 8,
  "auth.password_min_length": 14,
  "auth.allowed_methods": ["passkey", "totp"],
  "modules.enabled": ["calendar", "mail", "chat"],
  "modules.disabled": ["billing"],
  "naming.email_pattern": "{first}.{last}@{domain}",
  "naming.username_pattern": "{first[0]}{last}"
}

Question : Resolution de conflits entre policies

Quand 2 policies au meme niveau definissent la meme cle, qui gagne ?

A

Priorite numerique (comme AD GPO)

Chaque policy a un priority: number. Le plus petit nombre gagne. Previsible, facile a debugger. L'admin ordonne explicitement. + flag ENFORCED pour les policies non-blocables (le groupe impose).

B

Le plus strict gagne (comme Kanidm)

Pour chaque setting : le plus restrictif l'emporte. session=8h bat session=12h. mfa=true bat mfa=false. Simple, securitaire par defaut. Mais pas toujours previsible (qui decide ce qui est "plus strict" pour un pattern de nommage ?).

C

Hybride : strict pour securite + priorite pour le reste

Domaine security → le plus strict gagne toujours. Domaines naming, modules, delegation → priorite numerique. Combine securite forte et flexibilite administrative. Recommande.
