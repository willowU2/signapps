10 patterns Kanidm adoptes (sans copier le code)

1. Entry-Attribute-Value
Tout est une "entry" (personne, groupe, policy, noeud). Extensible sans migration.
2. Schema-as-data
Schema stocke en BDD, pas hardcode. Types d'attributs extensibles.
3. MemberOf auto-compute
Index inverse des groupes calcule automatiquement. Groupes imbriques aplatis.
4. Policy-on-groups
Politiques attachees aux groupes, resolution "le plus strict gagne".
5. Privilege elevation
Sessions longues read-only, re-auth pour ecrire. Fenetre courte.
6. High-privilege taint
Groupe "high_privilege" empeche les roles inferieurs de modifier les admins.
7. Recycle bin
Suppression douce → recyclage → tombstone. Recuperable par admin.
8. Event pipeline
Request → Event → Validation → ACP check → Plugins pre → Write → Plugins post.
9. Referential integrity
Plugin post-write verifie toutes les refs UUID. Nettoyage auto des liens morts.
10. Fail-fast batch
Operations batch atomiques : tout reussit ou tout echoue. Zero etat partiel.

Question : Modele de donnees

Comment structurer les entries dans PostgreSQL ?

A

Full EAV (comme Kanidm)

Une table entries + entry_attributes. Ultra-flexible, extensible sans migration. Mais queries complexes, pas de contraintes SQL natives, JOINs couteux.

B

Relationnel pur (actuel)

Tables dedicaces : org_nodes, persons, assignments, policies. Performant, contraintes FK, indexes. Mais chaque extension = migration SQL.

C

Hybride (recommande)

Tables relationnelles pour le coeur (noeuds, personnes, assignments, policies) + colonne JSONB attributes pour les extensions + pgvector pour le RAG. Le meilleur des deux mondes : performance SQL + flexibilite schema-less.
