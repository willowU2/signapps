3. Resolution de policy — Algorithme

resolve_effective_policy(person):
// 1. Collecter les policies par source
org_policies ← walk_ancestors(person.assignments[*].node_id)
site_policies ← walk_ancestors(person.site_assignments[*].site_id)
country_policies ← unique(person.sites[*].country_code)
group_policies ← person.memberof[*].policies
global_policies ← policies WHERE link_type = 'global'
// 2. Filtrer les desactivees et les bloquees
all ← filter(!disabled AND !blocked_at_ancestor)
// 3. Separer enforced / normales
enforced ← all WHERE is_enforced = true
normal ← all WHERE is_enforced = false
// 4. Merger par domaine
security: max_strict(enforced) THEN max_strict(normal)
modules, naming, delegation, compliance:
enforced(priority ASC) THEN normal(priority ASC)
custom: deep_merge(priority ASC)
// 5. Retourner policy effective + sources
return { settings, sources[] }

4. Patterns Kanidm appliques

✅ Entry lifecycle: Live → Recycled → Tombstone
✅ MemberOf auto-compute (trigger PG)
✅ High-privilege taint group
✅ Privilege elevation (re-auth pour ecrire)
✅ Referential integrity (FK + trigger cleanup)
✅ Fail-fast batch (transactions atomiques)
✅ Event pipeline (validate → ACP → write → post)
✅ Schema-as-data (org_node_types extensible)
✅ Policy-on-groups (GPO attachable partout)
✅ LDAP gateway read-only (futur)
