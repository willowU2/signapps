Types de groupes

Statique
Membres ajoutes manuellement par un admin. Ex: "Comite RGPD", "Projet Alpha".
Dynamique (filtre)
Membres calcules par un filtre sur les attributs. Ex: "All IT" = node_type=dsi OR node_type=engineering.
Derive de l'arbre
Auto-genere depuis la structure org. Ex: "Managers N-1" = toute personne holder d'un noeud enfant direct de la racine.
Hybride
Base dynamique + membres manuels en override. Ex: "All IT" dynamique + "Dave" ajoute manuellement car il fait du dev meme s'il est au commercial.

Capacites cles des groupes transverses

Capacite Description Exemple
Imbrication Un groupe peut contenir d'autres groupes "All Staff" contient "All IT" + "All HR" + ...
MemberOf auto Index inverse calcule automatiquement (pattern Kanidm) Alice.memberof = [All IT, Comite RGPD, DSI France]
Policy attachable Les GPO s'attachent aux groupes transverses aussi "All IT" → policy modules.devtools=true
Delegation Un groupe peut etre manage par un autre groupe "Projet Alpha" manage par "PMO Office"
Scope temporel Debut/fin optionnels (projets, missions) "Projet Alpha" actif du 01/04 au 31/12/2026

Question : Ou vivent les groupes transverses ?

Faut-il un espace dedie ou les integrer dans l'arbre ?

A

Entite separee — table dediee

Les groupes sont une entite premiere classe a cote des arbres. Table org_groups avec son propre CRUD. Ils apparaissent dans un onglet "Groupes" a cote de l'arbre. Separation claire, mais navigation fragmentee.

B

Dans l'arbre — comme un type de noeud special

Un groupe transverse est un OrgNode avec node_type=cross_group. Il vit dans un arbre dedie "Groupes transverses". Unifie le modele, mais melange hierarchie et groupes.

C

Hybride unifie (recommande)

Table dediee org_groups MAIS visible comme un panneau lateral dans la meme page. Les groupes peuvent referencer des noeuds (tous les enfants de ce noeud sont membres). Vue unifiee : arbre a gauche, groupes en overlay, detail a droite. Un seul endroit pour tout gerer.
