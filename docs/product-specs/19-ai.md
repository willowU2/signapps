# Module AI (Chat, Tools, Agents) — Spécification fonctionnelle

## Benchmark concurrentiel

| Outil | Forces distinctives à capturer |
|---|---|
| **ChatGPT (OpenAI)** | Référence grand public, GPT-4/4o, Custom GPTs (personnalisables), Memory (mémoire contextuelle), Code Interpreter, DALL-E image generation, voice conversation, vision, team workspaces, web browsing, file upload, canvas mode |
| **Claude (Anthropic)** | Context window massif (200k+), Artifacts (code/docs générés en side panel), Projects (collections avec knowledge), Computer use (bêta), prompt caching, API-first |
| **Gemini (Google)** | Intégration Workspace, multi-modal natif (texte, image, audio, vidéo), 1M+ context window, Gemini Pro/Ultra, Gems (custom assistants), intégration Search |
| **Perplexity** | Recherche web + LLM, sources citées, Pages (reports sauvegardables), Copilot (guided search), mobile, Spaces |
| **Poe (Quora)** | Multi-model chat (GPT, Claude, Gemini, Llama), bots custom, créator economy |
| **Character.ai** | Personnages AI avec personnalités, conversations persistantes |
| **Mistral Le Chat** | Français-natif, multi-model français, open source models |
| **Microsoft Copilot** | Intégration Office 365, tout-en-un, modèles Azure |
| **Replit AI** | Code-focused, pair programming, debugging |
| **GitHub Copilot** | Code completion in IDE, Copilot Chat, PR review |
| **Cursor** | VSCode fork AI-first, Composer, codebase-aware |
| **LM Studio / Ollama** | Local LLM, privacy-first, open source |
| **LibreChat** (open source) | Multi-provider chat interface, self-hosted |

## Principes directeurs

1. **Multi-model, multi-provider** — utilisateur choisit le modèle (Claude, GPT, Gemini, local) selon son besoin, sans vendor lock-in.
2. **Context-aware** — l'IA connaît le contexte SignApps de l'utilisateur (ses docs, ses contacts, ses tâches) avec consentement explicite.
3. **Tools intégrés** — l'IA peut agir : créer un événement, envoyer un email, lancer une recherche, générer un fichier.
4. **Local-first options** — modèles locaux pour la privacy absolue (whisper, llama.cpp, ollama integration).
5. **Conversations persistantes et organisées** — historique, favoris, tags, recherche.
6. **Transparency** — l'IA cite ses sources, indique quand elle n'est pas sûre, n'hallucine pas les fonctions.

## Catégorie 1 — Chat et conversations

### 1.1 Interface de chat
Vue centrale : messages utilisateur à droite, messages IA à gauche. Input en bas avec formatting riche. Sidebar gauche avec liste des conversations.

### 1.2 Nouvelle conversation
Bouton `+` pour démarrer une conversation. Optionnel : sélection du modèle, du system prompt, du contexte.

### 1.3 Sidebar des conversations
Liste chronologique des conversations passées. Chaque conversation a un titre auto-généré (résumé de la première question).

### 1.4 Renommer une conversation
Clic sur le titre → édition. Changement manuel si l'auto-title n'est pas bon.

### 1.5 Organisation des conversations
- **Folders / Projects** : grouper des conversations par sujet
- **Tags** : étiquettes multiples par conversation
- **Favoris / épinglage** en haut
- **Archive** : conversations masquées mais non supprimées

### 1.6 Recherche dans les conversations
Barre de recherche qui cherche dans le titre et le contenu de toutes les conversations. Indexée.

### 1.7 Export de conversations
Export en Markdown, JSON, PDF. Partage par lien.

### 1.8 Suppression
Supprimer une conversation → corbeille (30 jours) puis définitif.

### 1.9 Partage public
Partager une conversation par lien public (lecture seule). Option d'anonymisation.

### 1.10 Continue conversation
Les conversations sont persistantes : on peut reprendre plus tard, l'historique est complet.

### 1.11 Fork conversation
Créer un "fork" d'une conversation pour explorer une autre direction sans affecter l'original.

### 1.12 Regenerate response
Bouton pour régénérer une réponse de l'IA (si pas satisfaisante).

### 1.13 Edit user message
Modifier un message qu'on a envoyé et regénérer la réponse depuis ce point.

### 1.14 Copy response
Bouton copy pour copier rapidement la réponse dans le presse-papier.

### 1.15 Quote response
Citer une partie d'une réponse IA dans un nouveau message.

### 1.16 Thumbs up / down
Feedback rapide sur la qualité d'une réponse (pour l'apprentissage et l'amélioration).

### 1.17 Message actions
Menu contextuel : copier, citer, fork, report, edit, delete.

## Catégorie 2 — Modèles et providers

### 2.1 Multi-model selector
Sélecteur du modèle en haut de chaque conversation ou changeable par message. Models disponibles :
- **Anthropic** : Claude Opus 4, Sonnet 4, Haiku 4
- **OpenAI** : GPT-5, GPT-4o, GPT-4o-mini
- **Google** : Gemini 2 Pro, Gemini 2 Flash
- **Mistral** : Mistral Large, Medium, Small
- **Local (Ollama)** : Llama 3, Mistral 7B, Phi, Qwen, etc.

### 2.2 Paramètres de modèle
Advanced settings par conversation : temperature, max tokens, top-p, system prompt, tools enabled, memory on/off.

### 2.3 System prompt custom
Définir un system prompt spécifique pour une conversation ou un modèle sauvegardé (ex: "You are a helpful French legal expert").

### 2.4 Model comparison
Envoyer le même prompt à plusieurs modèles et comparer les réponses side-by-side.

### 2.5 Model routing automatique
L'utilisateur peut définir un modèle "auto" qui choisit le meilleur pour chaque question (questions simples → Haiku/GPT-4o-mini, complexes → Opus/GPT-5).

### 2.6 Fallback
Si un provider a un outage, fallback automatique vers un autre modèle compatible.

### 2.7 Token counter
Affichage en temps réel du nombre de tokens utilisés (input + output) et du coût estimé.

### 2.8 Usage tracking
Dashboard : tokens consommés, coûts, par modèle, par jour/semaine/mois. Quotas par utilisateur et organisation.

### 2.9 Self-hosted models
Support pour Ollama, LM Studio, vLLM, llama.cpp. Configuration de l'endpoint et du model name.

### 2.10 API keys management
Stockage sécurisé des clés API OpenAI/Anthropic/Google via le module Vault. Chiffrées.

### 2.11 BYO keys vs managed
Option pour les utilisateurs d'utiliser leurs propres clés API (cost-free for SignApps) ou d'utiliser la facturation SignApps centralisée.

### 2.12 Rate limiting per user
Admin peut limiter le nombre de requêtes par jour/mois par utilisateur.

### 2.13 Budget alerts
Alertes quand un utilisateur/orga s'approche du budget mensuel.

## Catégorie 3 — Multi-modal (image, voice, video)

### 3.1 Image input
Drag-drop ou upload d'images dans le chat. L'IA (avec vision) peut analyser, décrire, extraire du texte, comparer.

### 3.2 Image generation
Prompt "Génère une image de ...". Appel à DALL-E, Midjourney API, Stable Diffusion, ou local (Flux). Multiples options, resize, upscale.

### 3.3 Image editing
Upload + prompt "Modifier cette image pour ...". Utilise les models inpainting.

### 3.4 Voice input
Bouton micro pour dicter une question. Transcription (whisper) puis envoi à l'IA.

### 3.5 Voice conversation
Mode conversation vocale : parle à l'IA, elle répond vocalement (TTS). Comme ChatGPT Advanced Voice.

### 3.6 Voice clone (avec consentement)
Upload d'un sample vocal pour que l'IA réponde avec une voix similaire à celle de l'utilisateur. Respect RGPD.

### 3.7 Video input
Upload d'une vidéo courte → l'IA analyse le contenu (Gemini, GPT-4o).

### 3.8 Audio transcription
Upload d'un fichier audio → transcription + résumé. Multi-language.

### 3.9 PDF / Document understanding
Upload PDF, DOCX, XLSX → l'IA lit et répond à des questions sur le contenu.

### 3.10 Screenshot analysis
Coller une capture d'écran pour que l'IA explique ce qu'elle voit.

### 3.11 Code screenshot to code
Photo d'un écran de code → extraction du code en texte.

### 3.12 Document OCR
Scan d'un document papier → OCR + compréhension IA.

## Catégorie 4 — Tools et function calling

### 4.1 Tools disponibles
- **Web search** (Perplexity-style, Bing, Google, Brave)
- **Calculator** (calculs précis)
- **Code interpreter** (Python sandbox)
- **SignApps modules** (Mail, Calendar, Docs, Tasks, CRM, etc.)
- **File operations** (read, write, list)
- **HTTP requests**
- **Image generation**
- **Date/time**
- **Unit conversion**
- **Custom tools** (défini par l'utilisateur/admin)

### 4.2 Tools auto-selection
L'IA choisit automatiquement les tools pertinents pour répondre à une question (function calling natif).

### 4.3 Tools manuel
L'utilisateur peut forcer l'utilisation d'un tool spécifique ("Cherche sur le web", "Lance le code interpreter").

### 4.4 Tool output visible
L'utilisateur voit quel tool a été appelé, avec quels arguments, et le résultat brut. Transparence.

### 4.5 Tool approval
Pour les tools sensibles (ex: envoyer un email, supprimer un fichier), demande de confirmation avant exécution.

### 4.6 Custom tools par l'utilisateur
Définir un tool custom : nom, description, schéma d'arguments, endpoint (HTTP ou script local). L'IA peut l'utiliser.

### 4.7 MCP servers
Support du Model Context Protocol (MCP) pour connecter des serveurs externes comme sources de tools et contexte.

### 4.8 OpenAPI importer
Importer une spec OpenAPI pour créer automatiquement des tools à partir des endpoints.

### 4.9 Tool history
Historique des tools utilisés par l'IA avec l'exécution détaillée.

### 4.10 Tool permissions
Par rôle ou par tool, définir qui peut les utiliser.

## Catégorie 5 — Contexte et knowledge

### 5.1 Projects / Workspaces
Créer un "projet" qui contient des conversations, des documents uploadés, et un system prompt custom. L'IA du projet a accès à tous ces docs.

### 5.2 Knowledge base upload
Upload de documents (PDF, DOCX, MD, TXT) comme knowledge pour un projet. RAG automatique.

### 5.3 SignApps context access
Avec consentement, l'IA peut accéder au contexte SignApps de l'utilisateur :
- Ses docs récents
- Ses contacts
- Ses calendriers
- Ses emails (opt-in explicite)
- Ses tâches
- Ses messages chat

### 5.4 Memory
Memory persistante entre conversations : l'IA se souvient de faits importants sur l'utilisateur ("Je travaille chez Acme", "Mon manager est Jean"). Toggleable et effaçable.

### 5.5 Edit memory
Page dédiée pour voir et éditer la memory de l'IA. Ajouter, supprimer, modifier.

### 5.6 Forget / Clear memory
Bouton pour tout oublier. Utile avant partage public ou reset.

### 5.7 Citations
Quand l'IA utilise un document du knowledge, elle cite la source (nom du fichier + page/section).

### 5.8 Upload limit
Limites raisonnables d'upload (X MB par fichier, Y fichiers par projet) pour éviter les abus.

### 5.9 Context window management
L'IA affiche un indicator quand le context window approche de sa limite. Suggestion de démarrer une nouvelle conversation ou compresser l'historique.

### 5.10 Context summarization
Résumé automatique de l'historique de conversation pour libérer du context window.

## Catégorie 6 — Agents et workflows

### 6.1 Custom agents / Gems / GPTs
Créer un agent personnalisé : nom, description, avatar, system prompt, tools autorisés, knowledge files, exemples de conversations.

### 6.2 Agent gallery
Galerie publique d'agents créés par les utilisateurs de l'org (et optionnellement publics).

### 6.3 Ready-made agents
Agents prêts à l'emploi : "Email Writer", "Code Reviewer", "Meeting Summarizer", "Research Assistant", "French Translator".

### 6.4 Agent invocation
Démarrer une conversation avec un agent spécifique. Le context et les tools de l'agent sont activés.

### 6.5 Multi-agent orchestration
Plusieurs agents dans une conversation, chacun avec son rôle (ex: "Planner agent" + "Executor agent" + "Reviewer agent").

### 6.6 Agent workflows
Workflows où un agent exécute plusieurs tools et sub-tasks de manière autonome pour accomplir un objectif complex (ex: "Réserve-moi un vol pour Paris la semaine prochaine").

### 6.7 Long-running agents
Agents qui travaillent sur une tâche sur plusieurs minutes/heures (ex: "Analyse tous mes emails de la semaine et fais un rapport"). Status en live.

### 6.8 Agent memory
Chaque agent a sa propre memory distincte de l'user memory.

### 6.9 Agent training / fine-tuning
Option pour fine-tuner un agent sur des exemples spécifiques (few-shot learning dans le system prompt).

### 6.10 Agent sharing
Partager un agent avec l'équipe. Import/Export.

### 7.1 Code interpreter (sandbox Python)
Exécution de code Python dans un sandbox sécurisé. L'IA peut écrire, exécuter, et itérer sur du code pour résoudre des problèmes (maths, analyse de données, génération de graphiques).

Cellules exécutables avec state persistent entre cells. Visualisations matplotlib/plotly rendues inline.

Suggestions de code pendant que l'utilisateur tape (copilot-style). Dans l'input du chat si on tape du code, ou dans les fichiers uploadés.

### 7.5 Repository analysis
Upload d'un zip de projet ou connexion GitHub/GitLab → l'IA analyse le code et peut répondre à des questions.

Via tools : create branch, commit, push, PR review. Sous supervision.

Coller une stack trace → l'IA identifie la cause probable et propose un fix.

Sélectionner un bloc de code → "Explique ce code". Explanation en français.

### 7.9 Code refactor
"Refactor ce code pour être plus propre/performant/testable". Propositions avec diff.

### 7.10 Generate tests
À partir d'une fonction, génération des tests unitaires.

## Catégorie 11 — Sécurité et privacy

### 11.1 Data encryption
Toutes les conversations chiffrées at rest (AES-256) et in transit (TLS 1.3).

### 11.2 No training on user data
Par défaut, les conversations ne sont pas utilisées pour entraîner les modèles (sauf opt-in explicite).

### 11.3 Local models option
Pour la privacy absolue, utiliser des modèles locaux (Ollama, llama.cpp) qui ne sortent jamais de l'infra SignApps.

### 11.4 Data retention policy
Conversations conservées selon règles : X jours / X années / forever. Configurable par user et admin.

### 11.5 Conversation deletion
Suppression définitive sur demande. RGPD-compliant.

### 11.6 Audit logs
Log : qui a utilisé l'IA, avec quels prompts, quels résultats. Pour compliance enterprise.

### 11.7 PII redaction
Option d'automatiquement remplacer les données personnelles (emails, noms, téléphones) avant envoi au LLM externe.

### 11.8 Content filtering
Filtrage des contenus interdits (violence, illégal, discriminations). Refus de réponse avec explication.

### 11.9 Jailbreak detection
Detection des tentatives de contournement des safety guardrails. Notification admin.

### 11.10 SOC 2 / ISO 27001 compliance
Respect des standards enterprise pour la sécurité.

### 11.11 Data residency
Choix de la région de stockage (EU, US). Important pour RGPD.

### 11.12 API keys in Vault
Toutes les clés API stockées dans le Vault SignApps, jamais en clair.

### Pattern d'implémentation recommandé
1. **Chat UI** : Vercel AI SDK (Apache-2.0) pour le streaming, Tiptap (MIT) pour l'input rich.
2. **Provider abstraction** : pattern strategy avec des implémentations par provider (OpenAI, Anthropic, Google, Ollama). Interface unifiée.
3. **Vercel AI SDK** : déjà fait le job pour les messages, streaming, tool calls.
4. **Local models** : wrapper autour d'Ollama (MIT). Pattern `localhost:11434/api/chat`.
5. **RAG** : LlamaIndex (MIT) ou Haystack (Apache-2.0) pour l'indexation. pgvector ou Qdrant pour le stockage.
6. **Embeddings** : sentence-transformers (Apache-2.0) local, ou OpenAI/Google embeddings API.
7. **Tools / Function calling** : implémentation custom avec schemas JSON Schema. Standard MCP.
8. **Code interpreter** : container Docker avec Python + common libs, communication via stdin/stdout.
9. **Image generation** : API OpenAI DALL-E / Anthropic / local Flux via ComfyUI.
10. **Voice** : whisper-rs (Unlicense) pour STT, Piper TTS (MIT) pour TTS local, ou services cloud.
11. **Vision** : passage direct au provider vision-capable (Claude, GPT-4o, Gemini).
12. **PDF parsing** : pdf-parse (MIT) ou PyMuPDF (AGPL — attention).
13. **Web search** : Brave Search API (freemium), Tavily API (freemium), ou SearXNG (AGPL **INTERDIT**).
