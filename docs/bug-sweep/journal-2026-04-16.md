### Module storage + drive (2026-04-16)

Reviewer: feature-dev:code-reviewer subagent.
Findings: P0=2, P1=3, P2=2 (+ workspace-wide P2 bonus)

**Fixed:**
- **P0** `client/src/components/storage/file-preview-dialog.tsx:388` — Infinite
  reload loop. `loadContent` depended on `blobUrl`; each successful load
  called `setBlobUrl(url)` → deps change → callback re-created → useEffect
  re-fired → re-download. Repeated until dialog closed or memory exhausted.
  Added `blobUrlRef` for in-callback revocation; removed `blobUrl` from deps.
- **P0** `services/signapps-storage/src/handlers/files.rs:920` — `delete_many`
  batch endpoint bypassed per-file authorization. Any authenticated user could
  delete files they did not own by knowing bucket/key pairs. Added
  `check_file_permission(..., Action::delete())` inside the loop and switched
  from `Extension<Uuid>` to `Extension<Claims>`.
- **P1** `services/signapps-storage/src/handlers/files.rs:633` — Upload handler
  used raw filename as storage key → concurrent uploads of the same filename
  silently overwrote each other's content. Keys are now UUID-prefixed
  (`{uuid}/{filename}`). UploadResponse returns the prefixed key so the client
  can download/delete consistently. Download's Content-Disposition still
  extracts the correct filename via `split('/').next_back()`.
- **P1** `client/src/components/storage/drop-zone.tsx:200` — `onUploadComplete`
  callback was checking `uploads.every(...)` on a stale closure snapshot from
  `handleFiles`, never seeing the items the caller had just added. Moved the
  completion check into the XHR `load` handler against the fresh `setUploads`
  snapshot.
- **P1** `client/src/lib/api/storage.ts:14` — Removed duplicate `Bucket`
  interface (2 declarations merged by TS, making consumers blind to missing
  fields like `owner_id`/`size_bytes`/`object_count`). Kept the full version.
- **P2 (workspace bonus)** — Removed 123 files worth of duplicate
  `#[tracing::instrument(skip_all)]` copy-paste attributes across services/
  and crates/. Each handler was getting a double-nested span with identical
  names, polluting distributed traces. Mechanical fix via sed, cargo check
  clean.

**Skipped (follow-up):**
- `client/src/lib/api/drive.ts:123` `findNodeByTargetId` only scans root nodes.
  Requires a new backend search endpoint — deferred to dedicated refactor.
