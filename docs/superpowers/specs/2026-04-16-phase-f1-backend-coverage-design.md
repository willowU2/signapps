# Phase F1 — Backend Test Coverage — Design Spec

**Date:** 2026-04-16
**Statut:** Validated, ready for implementation plan
**Auteurs:** Brainstorming Claude + Étienne

---

## 1. Context & Goals

### 1.1 Problem

Le workspace Rust SignApps compte ~1980 tests (`#[test]` + `#[tokio::test]`) sur 40 workspace members, mais :
- `cargo-llvm-cov` est référencé par `just coverage` MAIS n'est PAS installé (`cargo llvm-cov` retourne "no such command")
- Aucun baseline de coverage n'est mesuré ni catalogué
- On ne sait pas quelles zones sécurité/crypto/auth sont sous-testées

La Phase A bug sweep a identifié plusieurs items sécurité deferred (XOR LDAP, JWT aud bypass, password policy inline). Ces zones manquent probablement de tests de régression.

### 1.2 Goals

1. **Installer** `cargo-llvm-cov`
2. **Générer** le baseline lcov.info workspace + catalog markdown par crate/service avec % coverage
3. **Identifier** les hot spots sécurité/crypto non couverts
4. **Ajouter** 4-6 tests ciblés de régression sur ces hot spots (issus de Phase A + audits)
5. **Documenter** le baseline dans le journal Phase E/F

### 1.3 Non-goals

- **Pas d'objectif de coverage globale** (pas de "80% target")
- **Pas de refactor** pour rendre du code testable (le code non-testable reste tel quel)
- **Pas de mutation testing** (`just mutants` existe, pas touché)
- **Pas d'intégration CI** pour bloquer sur un drop de coverage (Phase F3 si nécessaire)
- **Pas de tests d'intégration nouveaux** contre DB réelle (seulement tests unitaires in-memory quand possible)

## 2. Paramètres validés

| Axe | Choix |
|---|---|
| Exécution baseline | **Dans la session** — setup → baseline → catalog → fixes enchaînés |
| Scope ajouts | **4-6 tests** sur hot spots sécurité/crypto identifiés par le catalog |
| Gouvernance | Auto-chain, validation auto par `nextest` après chaque commit |

## 3. Architecture du workflow

### 3.1 Flow

```
cargo install cargo-llvm-cov
  ↓
just coverage  (cargo llvm-cov --workspace --lcov --output-path lcov.info)
  ↓ 10-30 min
lcov.info (raw, ~20MB)
  ↓
tools/coverage-catalog.mjs  (parse lcov, group by crate, compute %)
  ↓
docs/bug-sweep/coverage-catalog.md  (human-readable, priorized)
  ↓
identify hot spots → add 4-6 tests → re-run nextest + llvm-cov
  ↓
docs/bug-sweep/coverage-catalog.md  (refreshed — delta committed)
```

### 3.2 Composants

| # | Fichier | Rôle |
|---|---|---|
| 1 | `Cargo.lock` | cargo-llvm-cov install (changes lock only if already a direct dep) |
| 2 | `tools/coverage-catalog.mjs` | Node script : parse lcov.info → markdown |
| 3 | `docs/bug-sweep/coverage-baseline.lcov` | Raw lcov, committed for diff next time |
| 4 | `docs/bug-sweep/coverage-catalog.md` | Grouped by crate/service, % coverage, TOP 10 low-coverage files |
| 5 | 4-6 nouveaux tests `#[test]` / `#[tokio::test]` | Co-localisés (`#[cfg(test)] mod tests` dans le même fichier) |

### 3.3 Format du catalog

```markdown
# Backend Coverage Catalog — 2026-04-16

Source: lcov.info generated by cargo-llvm-cov
Total lines: <L> | Covered: <C> | Coverage: <%>

## By workspace member (sorted by coverage ascending)

| Member | Type | Lines | Covered | Coverage |
|---|---|---|---|---|
| signapps-X | crate/service | 2450 | 123 | 5.0% |
| ...

## Top 20 lowest-coverage files (security-sensitive first)

| File | Lines | Covered | Coverage | Note |
|---|---|---|---|---|
| crates/signapps-keystore/src/keystore.rs | 320 | 12 | 3.8% | Crypto — HIGH PRIORITY |
| ...

## Security/crypto hot spots recommended for F1 additions

1. `crates/signapps-common/src/auth.rs` — JWT verify_token
2. `crates/signapps-keystore/src/*` — encrypt_string / decrypt_string
3. `services/signapps-identity/src/password_policy.rs` — validate
...
```

### 3.4 Stratégie de sélection des tests

Les 4-6 tests ajoutés viennent de **l'intersection** :
- Hot spot identifié par le catalog (faible coverage sur fichier critique)
- Issue touchée par Phase A (régression probable / cas limite non testé)

**Candidats concrets** (à vérifier contre le catalog) :

1. **`crates/signapps-common/src/auth.rs`** — test `verify_token` avec mauvaise audience (Phase A P0 fix). Vérifie que le token signé pour un autre audience est rejeté après le fix.
2. **`crates/signapps-keystore/src/*`** — test encrypt/decrypt roundtrip avec DEK rotation. Crypto critique.
3. **`services/signapps-identity/src/password_policy.rs`** — test `validate` avec cas edge : password all-lowercase, < 8 chars, sans digit, etc. (Phase A P1 fix).
4. **`services/signapps-identity/src/handlers/auth.rs`** — test `password_reset_confirm` que mon fix Phase A est respecté (uppercase + lowercase + digit required).
5. **`crates/signapps-rate-limit/src/lib.rs`** — test XFF parsing avec multiple proxies + token injection (Phase A deferred item).
6. **`services/signapps-storage/src/handlers/files.rs`** — test `delete_many` ACL per-key (Phase A P0 fix — assurer qu'un user sans permission est rejeté).

On choisit 4 parmi les 6 selon ce que le catalog révèle de critique.

## 4. Plan d'exécution (7 commits estimés)

1. **`chore(coverage): install cargo-llvm-cov`** (changes Cargo.lock if direct dep; sinon just notes via `cargo install --locked`)
2. **`tools(coverage): add coverage-catalog.mjs script`** — parse lcov, emit markdown
3. **`chore(coverage): generate backend baseline`** — run `just coverage`, commit `coverage-baseline.lcov` + `coverage-catalog.md`
4-7. **`test(<crate>): add regression test for <hot spot>`** — 4 commits, 1 par test

Chaque commit de test :
- `cargo nextest run -p <crate>` passe (le nouveau test inclus)
- Si le test expose un vrai bug (pas juste coverage), on fix + second commit

---

## 5. Validation

Par commit :
- `cargo check --workspace` 0 erreurs
- `cargo nextest run -p <crate>` 0 échecs
- Après commits test : re-run `cargo llvm-cov --lcov -p <crate> --output-path /tmp/cov-delta.lcov` — le delta doit montrer les lignes maintenant couvertes

Final (Task 7+) :
- Catalog regénéré, `coverage-catalog.md` mis à jour pour refléter les ajouts
- % global noté dans le journal (delta avant/après)

---

## 6. Error handling

- **cargo-llvm-cov install fail** : fallback possible — `cargo install --locked cargo-llvm-cov` si hit sur MSVC Windows. Si échec persistant, déferre via tarball prebuilt binary ou rustup component.
- **`just coverage` échoue** sur un crate spécifique : commenter l'entrée fautive dans une ignore-list temporaire, noter dans `bug-sweep-todo.md`.
- **Test nouveau expose un bug** : créer un commit `fix(<crate>): <bug>` + le test dans un autre commit.

---

## 7. Testing

Les tests de cette phase sont les livrables eux-mêmes. Pas de meta-tests.

---

## 8. Success criteria

- [ ] `cargo llvm-cov --version` affiche une version ≥ 0.6
- [ ] `just coverage` s'exécute end-to-end sans échec
- [ ] `docs/bug-sweep/coverage-catalog.md` généré avec au moins :
  - Total workspace % coverage
  - Liste des workspace members avec leur % coverage
  - Top 20 files à faible coverage
  - Section "Security/crypto hot spots"
- [ ] 4 tests minimum ajoutés (hot spots sécurité/crypto)
- [ ] Chaque test couvre une ligne auparavant non couverte (vérifié par delta coverage)
- [ ] `cargo nextest run --workspace` passe (aucune régression)

## 10. Glossaire

- **cargo-llvm-cov** : plugin cargo qui instrumente les tests via `-C instrument-coverage`, collecte des profraw, produit du lcov/html.
- **lcov.info** : format standard LCOV, consommable par Codecov, genhtml, ou scripts custom.
- **Hot spot** : fichier à fort risque (sécurité/crypto/auth) avec faible coverage.
- **Régression test** : test écrit après un bug fix, qui échoue sur le code pré-fix et passe sur le code post-fix.
